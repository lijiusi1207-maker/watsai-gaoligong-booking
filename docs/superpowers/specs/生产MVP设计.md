# 哇噻·高黎贡预约系统生产 MVP 设计

## 目标

把现有演示版推进到可小范围真实运营的 MVP：公网部署就绪、账号与管理端具备基本安全、预约有容量和取消闭环、家庭成员可挂在一次订单下、凭证可被扫码读取，并保留零依赖 Node 的低运维特性。

## 方案

继续使用现有 `服务器.js` 单体服务，不引入大型框架。数据结构向后兼容：旧单人预约自动视为一个 participant，新增 `participants`、`cancelledAt`、`sessionKey`、`authToken` 字段；`data.json` 仍仅作为 MVP 数据层，但生产说明明确其不适合规模化，后续迁移数据库。

账号登录采用基于 `crypto.scryptSync` 的密码哈希；后台 token 读取 `process.env.ADMIN_TOKEN`，开发环境保留显式警告而不再把生产口令写死。用户登录返回短期随机 session token，敏感接口优先接受该 token；为兼容现有前端，uid 查询暂时保留并限制为公开脱敏数据。

预约接口按活动和场次计算容量，拒绝重复有效预约；支持取消并释放名额。家庭成员不创建独立账号，预约时以 participant 列表保存姓名、年龄、监护关系和身份证脱敏信息，用户本人仍是责任主体。凭证码继续使用现有 YX-/MP- 前缀，并新增可扫码 URL/文本 payload，后台核销接口复用同一凭证校验逻辑。

## 验证

- 单元/接口回归：密码不落明文、错误 token 被拒、容量超限被拒、重复预约被拒、取消释放名额、家庭成员随订单返回、凭证 payload 可解析。
- 现有 `回归验证.js`、`新功能验证.js` 必须继续通过；测试使用临时数据文件，不能污染运营数据。
- `/healthz`、本机启动、Render 配置、生产环境变量和备份恢复说明必须可核验。

## 边界

本阶段不接微信支付、微信订阅消息、微信原生扫码硬件、短信供应商或微信 openid；这些需要外部资质/账号，完成接口边界和配置说明即可。公网真正上线仍需要用户的 GitHub/Render/VPS 权限。
